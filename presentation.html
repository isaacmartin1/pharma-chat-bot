KEY DESIGN DECISION #1
The Claims Approval Gate

AI proposes claims. Human approves. Generation only happens after explicit sign-off. This eliminates hallucinated efficacy data at the source.

WITHOUT THIS GATE:
✗ AI invents efficacy numbers
✗ Hallucinated trial citations
✗ 21 CFR 202 violation risk
WITH THE GATE:
✓ Only library claims used
✓ Every claim has source citation
✓ Human approves before any generation

The Compliance Pipeline

6 layers · green / yellow / red · auto-fix loop · export gate
1 — Legal Rules 238 rules

FDA 21 CFR 202, OPDP, FDCA, state false-advertising (53 jurisdictions), PhRMA/industry codes. Each rule ID maps to a dedicated check function.

2 — Evidence Check citation tracing

Claims embedded as <!-- CLAIM_ID:xxx --> comments are traced to evidence records in the DB and verified for source accuracy.

3 — Brand Wording exact text match

Claim IDs verified against the approved claims library. No paraphrasing — the exact approved text must match. Red flag if a claim ID isn't found.

4 — Visual Assets + ISI blocking

All img src URLs checked against brand asset library. <!-- ISI_PRESENT --> marker checked — red failure if ISI footer missing.

5 — Channel Compatibility

HTML >100KB (email clipping), external CSS link, <script> tags, missing alt text, <style> blocks — email client compatibility checks.

6 — Grammar & Spelling

Repeated words, double spaces, sentences starting lowercase. Regex scan over stripped text — fast, no API call.

Why rules-as-data beats rules-as-prompts
❌ LLM evaluates rules at runtime
  • Non-deterministic — same HTML may pass/fail differently
  • Expensive — every check = API call (~$0.01 each)
  • Not auditable — can't show which specific rule failed
  • Hallucination risk in rule interpretation
✓ Rules seeded to DB, evaluated with Python functions
  • Deterministic — same HTML always gives same result
  • Free — pure regex/logic, zero API calls
  • Auditable — rule ID + name + message surfaced per check
  • Updateable — activate/deactivate rules without code change

Design Thinking: 4 Questions to Answer

These are explicit discussion questions in the exercise brief — every one has a concrete answer
🛑 Q1: How do you prevent non-compliant content?
Dual strategy — UI affordances + backend validation
  • Claims gate: user cannot trigger generation without explicitly approving library claims — the button literally doesn't exist until approval happens
  • Export block: red compliance failures disable the export button — compliance isn't optional
  • Real-time: 238-rule check fires immediately after every asset create/update, auto-fix retries ×2
  • Manual re-check: user can re-run the full suite at any time via "Run Checks" button
🤖 Q2: Role of AI vs. templates?
LLM for narrative + structure · constrained to approved content
  • Why not templates: HCP × patient × caregiver × 4 tones × 4 goals = 48 variants. Templates don't scale; they also can't handle iterative NL refinement
  • LLM generates: layout, narrative flow, brand voice, formatting, ISI copy
  • LLM is constrained by: only claims from the approved library can be proposed (gate enforces this), ISI marker required, output validated against 238 rules post-generation
  • Template-like for: channel scaffolding (email boilerplate, banner dimension constraints passed in system prompt)
📜 Q3: Content lineage after 15 edits?
Full HTML snapshot on every write · source field · user ID
  • Every write (AI or direct) snapshots previous full HTML + source ("ai_edit" / "direct_edit" / "undo") + changed_by user ID
  • Undo is O(1) — restore latest snapshot, no replay needed
  • Granularity: per-save, not per-keystroke — right level for MLR audit workflows
  • No branching yet — linear history. A/B variants would require a fork model (separate asset rows linked by parent_id)
  • Production: add SHA-256 content hash per version for tamper detection
👥 Q4: Concurrent editing — now vs. future?
NOW: last-write-wins · FUTURE: optimistic locking + WebSockets
  • Right now: last write wins. Race condition is possible if two users submit edits within the same second
  • Mitigation already in schema: version int on Asset — PUT /assets/{id} can reject writes where client's version ≠ DB version (optimistic locking, one-line add)
  • Real pharma reality: reviewer ≠ editor in MLR workflows — concurrent write risk is lower than it sounds
  • Next sprint: WebSocket presence indicators (see who's viewing), operational transform, conflict resolution UI
⚡ Performance Optimization Strategies
POC — implemented now
  • 5-min TTL cache on legal rules — avoids re-querying 238 rules on every compliance check
  • Async/await throughout FastAPI — generation and compliance don't block the event loop
  • 45s timeout on asset generation — prevents hung requests; user sees graceful error + retry
  • Sequential DB checks (necessary: SQLAlchemy sessions aren't thread-safe with SQLite)
Production path
  • SQLite → PostgreSQL + PgBouncer enables parallel compliance checks with thread-safe sessions
  • In-memory cache → Redis (shared across workers, TTL managed centrally)
  • Asset HTML in DB TEXT column → S3/blob for large slide decks (>100KB)
  • CDN for brand assets (currently served from FastAPI /static)
